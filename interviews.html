📘 JavaScript / TypeScript

var, let and const? var is function‑scoped, hoisted and initialized with undefined. It can be redeclared and updated, which can lead to bugs. let is block‑scoped, hoisted but in the Temporal Dead Zone (TDZ) until initialized; it can be updated but not redeclared in the same scope. const is block‑scoped, hoisted, and cannot be reassigned (though objects/arrays can be mutated). Always prefer const by default, and let when you need to reassign.
Hoisting? Hoisting is JavaScript's default behavior of moving declarations to the top of their containing scope during compilation. var declarations are hoisted with undefined, so you can access them before declaration (but get undefined). let and const are hoisted but not initialized — accessing them before declaration throws a ReferenceError due to the TDZ. Function declarations are hoisted entirely, so they can be called before defined.
Closure? A closure is a function that remembers its lexical scope even when the function is executed outside that scope. For example, an inner function that accesses variables from an outer function after the outer function has returned. Closures are used for data privacy, factory functions, and in event handlers/callbacks.
Event Loop? The Event Loop is the mechanism that enables asynchronous operations in JavaScript. It continuously checks the call stack and the task queues. When the call stack is empty, it takes tasks from the microtask queue (Promises, queueMicrotask) first, then from the macrotask queue (setTimeout, I/O). This non‑blocking model allows JavaScript to handle concurrency despite being single‑threaded.
Promise vs async/await? Promises are objects representing eventual completion or failure; they use .then() and .catch() for chaining. async/await is syntactic sugar over Promises, making asynchronous code look synchronous — easier to read and debug. Under the hood, async functions return a Promise, and await pauses execution until the Promise resolves. Error handling is done with try/catch instead of .catch().
Call Stack? The Call Stack is a LIFO (Last‑In‑First‑Out) data structure that tracks function calls. Each time a function is invoked, it is pushed onto the stack; when it returns, it is popped. It records exactly where we are in the program. If the stack grows too large (e.g., infinite recursion), a RangeError is thrown.
Why TypeScript? TypeScript adds static typing to JavaScript, catching errors at compile time rather than at runtime. It provides better IDE support (autocomplete, refactoring, navigation), improves code readability and maintainability in large codebases, and enables modern JavaScript features (ES6+) with backward compatibility. It also makes refactoring safer and reduces the need for extensive unit tests for type‑related bugs.
Interface vs Type? Interface is best for defining object shapes and is extendable via declaration merging (you can define the same interface multiple times and they combine). Type is more flexible — it can define unions, intersections, primitives, tuples, and mapped types, but it cannot be extended after declaration (no declaration merging). Use interface for public APIs and object definitions; use type for complex types or union/intersection scenarios.
Generic? Generics allow you to create reusable components that work with a variety of types while maintaining type safety. For example, function identity<T>(arg: T): T { return arg; } preserves the type of the argument. Generics are widely used in React components, Promise types, Array methods, and utility functions. They enable abstraction without losing type information.
unknown vs any? any disables all type checking — the compiler allows any operation on it, which defeats the purpose of TypeScript. unknown is a type‑safe counterpart to any: you cannot perform operations on an unknown value until you've narrowed it using type guards (typeof, instanceof, or custom checks). This forces you to handle the type properly, making your code safer.
never? never represents values that never occur. It is the return type of functions that always throw an error or have infinite loops. It is also used in exhaustive type checking — ensuring all possible cases are handled in a union type. never is a subtype of every type, but no type is a subtype of never.
Utility Types? Partial: makes all properties optional. Required: makes all properties required. Pick<T, K>: selects a subset of properties. Omit<T, K>: removes properties. Record<K, T>: defines a key‑value pair type. Readonly: makes all properties read‑only. These are built‑in mapped types that help transform existing types without duplication.

🔄 Proxy in JavaScript

What is a Proxy in JavaScript? A Proxy is an object that wraps another object (target) and intercepts fundamental operations on it, such as property access, assignment, enumeration, function invocation, etc. It allows you to define custom behavior for these operations using traps (handler methods). Proxies are part of ES6 and are used for meta‑programming, validation, logging, and reactive programming.
What are Proxy traps? Traps are handler methods that intercept operations on the target object. Common traps include: get (property access), set (property assignment), has (in operator), deleteProperty (delete), apply (function call), construct (new), ownKeys (Object.keys()), and getPrototypeOf / setPrototypeOf. Each trap allows you to customize the default behavior.
How do you create a Proxy? You create a Proxy with new Proxy(target, handler) where target is the object being proxied, and handler is an object containing traps. Example: const p = new Proxy({}, { get: (obj, prop) => prop in obj ? obj[prop] : 'default' }); This returns 'default' for missing properties.
Use cases for Proxy? Proxies are used for: validation (validate data before setting), logging (log property access), reactive programming (like Vue.js reactivity), memoization (cache results), access control (restrict property access), revocable proxies (temporarily disable access), and API mocking or virtual objects (lazy loading).
What is a revocable Proxy? A revocable Proxy is created with Proxy.revocable(target, handler). It returns an object with two properties: proxy (the Proxy instance) and revoke (a function). When revoke() is called, the proxy is completely disabled — any operation on it will throw a TypeError. This is useful for when you need to grant temporary access and then revoke it.
Proxy vs. Reflect API? Proxy intercepts operations, while Reflect is a built‑in object that provides default behavior for the same operations. They are often used together — inside a trap, you call Reflect.get(), Reflect.set(), etc. to implement the default behavior. This combination allows you to add custom logic while still preserving the original behavior when needed.
Can you proxy a function? Yes, you can proxy functions using the apply and construct traps. apply intercepts function calls, and construct intercepts new calls. This allows you to add logging, caching, or argument validation to functions. Example: new Proxy(() => {}, { apply: (target, thisArg, args) => { console.log('called'); return target(...args); } });
What are the limitations of Proxy? Proxies have some limitations: performance overhead (they are slower than direct access), cannot proxy all operations (e.g., == comparison falls back to the target), not available in older browsers (requires ES6), and no native support for proxying built‑in objects like Date or Array fully (some internal slots are not intercepted). Also, Object.isFrozen() and Object.seal() on a proxy affect the target, not the proxy itself.
How does Vue.js use Proxies? Vue.js 3 uses Proxies for its reactivity system. Instead of using Object.defineProperty (Vue 2), Vue 3 creates a Proxy that intercepts get and set operations on reactive objects. This allows tracking dependencies when properties are accessed (get) and triggering updates when properties are assigned (set). Proxies also handle dynamic property addition and deletion, which was a limitation in Vue 2.

🟢 Nuxt & Vue

What is Nuxt? Nuxt is a Vue.js framework that provides server‑side rendering (SSR), static site generation (SSG), automatic routing, and a modular architecture. It's similar to Next.js but for Vue. Nuxt 3 is built on Vue 3 and supports the Composition API, TypeScript, and the Nitro server engine, which is cross‑platform (Node.js, Deno, Cloudflare Workers, etc.).
What are Nuxt's rendering modes? Nuxt supports three main rendering modes: SSR (Server‑Side Rendering) — pages are rendered on the server per request; SSG (Static Site Generation) — pages are pre‑rendered at build time; and CSR (Client‑Side Rendering) — pages are rendered in the browser. Nuxt 3 also introduces Hybrid Rendering where you can mix modes per route using the routeRules in nuxt.config.ts.
What is the Nuxt App Router? Nuxt uses a file‑based routing system inside the pages/ directory. Each .vue file becomes a route. Nuxt 3 supports dynamic routes ([id].vue), catch‑all routes ([...slug].vue), and nested routes using directories. It also supports layouts (layouts/), middleware for route guards, and pages with definePageMeta for per‑page configuration.
What is the difference between Nuxt and Next? Nuxt is built on Vue.js and Next is built on React. Both offer SSR, SSG, file‑based routing, and API routes. Nuxt uses the Nitro server engine (which is more universal), while Next uses the Node.js runtime and the new App Router. Nuxt 3's reactivity is based on Vue's Proxy system, while Next uses React Server Components. The choice often comes down to preference between Vue and React.
What is Vue 3's Composition API? The Composition API is a set of APIs introduced in Vue 3 that allow you to organize component logic using functions rather than options (data, methods, computed, etc.). It uses setup() or the <script setup> syntax, and core functions include ref (reactive values), reactive (reactive objects), computed, watch, onMounted, and provide/inject. It enables better code reuse and organization.
What is <script setup> in Vue? <script setup> is a syntactic sugar for using the Composition API in Vue Single‑File Components (SFCs). It reduces boilerplate — you don't need to export setup(), and top‑level variables are automatically exposed to the template. It also provides better performance by avoiding the overhead of the setup() function wrapper. It's the recommended way to write Vue components in Nuxt 3.
How does Nuxt handle data fetching? Nuxt provides composables for data fetching: useAsyncData and useFetch. useFetch is a wrapper around useAsyncData and $fetch (from ofetch) that provides automatic caching, key generation, and error handling. These composables work on both server and client, and they automatically deduplicate requests. Nuxt 3 also supports useLazyFetch and useLazyAsyncData for client‑only loading with pending state.
What is a Nuxt module? A Nuxt module is a package that extends Nuxt's functionality by adding features like authentication (Nuxt Auth), PWA support, analytics, CMS integration, etc. Modules are installed via npm and added to the modules array in nuxt.config.ts. They can hook into Nuxt's lifecycle and modify the build process, add server routes, or inject composables.
What is the difference between ref and reactive in Vue? ref is used for primitive values and objects, and it wraps the value in an object with a .value property. reactive is used for objects and arrays only and returns a reactive proxy directly. ref is more flexible because it works with any type, and it's required for primitives. reactive doesn't need .value, but it cannot be destructured without losing reactivity (use toRefs).
What is the computed function in Vue? computed creates a reactive derived state that is cached based on its dependencies. It only re‑evaluates when its dependencies change, making it efficient for expensive calculations. In the Options API, it's defined inside the computed object; in the Composition API, it's used as a function: const double = computed(() => count.value * 2). It's read‑only by default unless you pass an object with get and set.
What is the watch function in Vue? watch is used to react to changes in reactive data. It takes a source (a ref, reactive object, or getter function) and a callback that runs when the source changes. Options include deep (for deep objects), immediate (run immediately), and flush (timing). watchEffect is a simpler version that automatically tracks dependencies and runs immediately.
How does Nuxt handle authentication? Nuxt commonly uses @nuxtjs/auth-next module or the newer @sidebase/nuxt-auth for authentication. These integrate with OAuth providers, JWT, and session‑based auth. They provide composables like useSession(), signIn(), signOut(), and middleware for protecting routes. In Nuxt 3, you can also implement custom authentication using Nitro server routes with JWT and cookies.
What is the Nuxt server/ directory? The server/ directory in Nuxt 3 is where you define server‑side logic. It includes API routes (server/api/), server middleware (server/middleware/), and server plugins (server/plugins/). This allows you to build a full‑stack application with a single codebase, with type‑safe communication between client and server using useFetch and $fetch.
What are Nuxt composables? Nuxt composables are reusable functions that encapsulate reactive logic. They are stored in the composables/ directory and are auto‑imported. Examples include useState (reactive shared state), useRouter (navigation), useRoute (route info), useHead (SEO), and the data‑fetching composables. They leverage Vue's Composition API and Nuxt's SSR capabilities.
What is SSR vs. SSG in Nuxt? SSR (Server‑Side Rendering) renders pages on the server for each request — good for dynamic, personalized content. SSG (Static Site Generation) pre‑renders pages at build time — faster and cheaper for static content. Nuxt allows you to use both with the ssr and target options. Nuxt 3's routeRules lets you define different rendering modes per route: ssr: false for CSR, static: true for SSG.
How does Nuxt handle SEO? Nuxt provides built‑in SEO features: useHead composable for setting meta tags, titles, and link tags. You can define it per page, in layouts, or globally in nuxt.config.ts. Nuxt also supports Open Graph, JSON‑LD (structured data), and sitemaps. For SSR, the meta tags are rendered on the server, ensuring search engines see the complete content.
What are Nuxt plugins? Nuxt plugins are files that run before your application is created. They are stored in the plugins/ directory and are auto‑imported. They can be used to: register Vue components globally, add third‑party libraries (e.g., Vue Toast), inject values into the app (provide), or run logic that should happen once on startup. You can set ssr: false to run them only on the client side.
What is the nitro server? Nitro is the server engine used by Nuxt 3. It's a universal server that can run on Node.js, Deno, Bun, Cloudflare Workers, and more. It supports file‑based API routes, server middleware, caching, and event‑driven architecture. Nitro also provides built‑in support for server pre‑rendering, serverless deployments, and static hosting (via output/).

🟢 Vue.js

What is Vue.js? Vue.js is a progressive JavaScript framework for building user interfaces. It is designed to be incrementally adoptable — you can use it as a library for specific features or as a full‑framework for building entire applications. Vue focuses on the view layer and provides reactive data binding, a component‑based architecture, and a virtual DOM for performance. It's known for its gentle learning curve, clear documentation, and versatility.
What are Vue's key features? Vue's core features include: Reactivity system (using Proxies in Vue 3), component‑based architecture (reusable, isolated components), template syntax (HTML‑based with directives like v-if, v-for, v-model), Composition API (for better code organization), single‑file components (.vue files), routing (via Vue Router), state management (via Pinia), transitions & animations, and server‑side rendering (via Nuxt or Vue SSR).
What is the difference between Vue 2 and Vue 3? Vue 3 introduced several major changes: Proxy‑based reactivity (replacing Object.defineProperty), which supports dynamic property addition and better performance. It introduced the Composition API for better logic reuse and organization. Vue 3 has faster rendering (optimized virtual DOM), smaller bundle size (tree‑shaking support), TypeScript support (written in TS), fragments (multiple root nodes), Teleport (render components outside the DOM hierarchy), Suspense (async components), and improved custom renderer API.
Options API vs Composition API? Options API is the classic way to define components using options like data, methods, computed, watch, and lifecycle hooks. It's simple and intuitive for small to medium components. Composition API uses functions (setup() or <script setup>) and allows better logic reuse, cleaner code organization for complex components, and improved TypeScript inference. It also enables reusable composables (custom hooks). Both APIs can be used together.
What is a Single‑File Component (SFC)? A Single‑File Component (.vue file) is a self‑contained component that encapsulates the template, script, and styles in one file. It uses <template> for HTML, <script> for JavaScript/TypeScript logic, and <style> for CSS (scoped automatically with scoped attribute). SFCs make components easier to maintain, test, and reuse. They are the recommended way to build Vue applications using build tools like Vite or Webpack.
What are Vue directives? Directives are special attributes with the v- prefix that apply reactive behavior to the DOM. Common built‑in directives: v-if (conditional rendering), v-for (list rendering), v-model (two‑way binding), v-bind (attribute binding, shorthand :), v-on (event handling, shorthand @), v-show (toggle visibility via CSS), v-html (render raw HTML, careful with XSS), and v-cloak (hide un‑compiled templates). You can also create custom directives.
How does Vue's reactivity system work in Vue 3? Vue 3's reactivity is built on JavaScript Proxies. When you create a reactive object with reactive() or ref(), Vue wraps it in a Proxy that intercepts get and set operations. During component rendering, Vue tracks which reactive properties are accessed (via the get trap) and collects dependencies. When a property changes (via the set trap), Vue schedules updates for all dependent components. This system is more efficient and handles dynamic properties and arrays better than Vue 2's Object.defineProperty.
What is the ref function in Vue? ref is a function that creates a reactive reference to a value. It wraps the value in an object with a .value property. ref works with any type — primitives, objects, arrays. In the template, ref values are automatically unwrapped, so you don't need .value. For example: const count = ref(0), then in template: {{ count }}. In script, you access it with count.value. ref is the recommended way to create reactive state in Vue 3.
What is the reactive function in Vue? reactive creates a deeply reactive object using a Proxy. It only works for objects and arrays (not primitives). Unlike ref, it doesn't require .value — you access properties directly. Example: const state = reactive({ count: 0 }). However, reactive cannot be destructured without losing reactivity — use toRefs() to convert each property to a ref. reactive is useful for complex objects, but ref is more versatile and recommended for most cases.
What is the difference between ref and reactive? ref works with any type, including primitives, and wraps the value in a .value object. It's more flexible and can be passed around easily. reactive only works with objects/arrays, doesn't need .value, but cannot be destructured without losing reactivity. ref is preferred for consistency, and you can use reactive for complex objects when you don't want to access via .value. Vue 3 also provides shallowRef and shallowReactive for performance optimization.
What is computed in Vue? computed creates a cached derived state that automatically updates when its dependencies change. It's used for values that are derived from other reactive data. Example: const double = computed(() => count.value * 2). Computed properties are lazily evaluated (only when accessed) and cached until dependencies change. They can also be writable by providing an object with get and set functions. In the Options API, they are defined in the computed object.
What are watch and watchEffect? watch observes a specific reactive source and runs a callback when it changes. It gives you access to the old and new values. Options include deep, immediate, and flush. watchEffect runs a function immediately and automatically tracks all reactive dependencies used inside it. It runs whenever any of those dependencies change. watchEffect is simpler but less precise. Both return a stop function to cancel the watcher.
What is the lifecycle of a Vue component? Vue components have a lifecycle with hooks you can use: Creation (setup(), beforeCreate, created), Mounting (beforeMount, mounted), Updating (beforeUpdate, updated), Unmounting (beforeUnmount, unmounted). In the Composition API, these are accessible as onMounted(), onUpdated(), onBeforeUnmount(), etc. setup() runs before beforeCreate — this is where reactive state is initialized.
What are Vue slots? Slots are a mechanism for content distribution in Vue. They allow a parent component to pass content (templates or markup) into a child component's template. There are default slots, named slots (<slot name="header" />), and scoped slots (where the child component passes data back to the parent). Slots enable flexible, reusable components like cards, modals, or data tables.
What are Vue's provide and inject? provide and inject are used for dependency injection — they allow a parent component to pass data to any descendant component, regardless of how deep it is, without using props. This is useful for global data like theme, user authentication, or configuration. In Vue 3, they are used with provide() in the parent's setup() and inject() in the child's setup(). The data is reactive if you provide a ref or reactive object.
What is Pinia? Pinia is the official state management library for Vue, replacing Vuex. It's simpler, more type‑safe (built with TypeScript), and supports the Composition API. Pinia stores are defined with defineStore() and can contain state, getters, and actions. It supports devtools, hot module replacement, and works with both Vue 2 and Vue 3. Pinia is the recommended state management solution for new Vue projects.
What is the difference between Vuex and Pinia? Vuex is the older state management library with mutations, actions, and modules. It has more boilerplate and limited TypeScript support. Pinia is the newer, official alternative that removes mutations (directly mutate state), has better TypeScript inference, supports multiple stores, and has a simpler API. Pinia also supports the Composition API and is more modular. Pinia is now the recommended choice for Vue 3 projects, and Vuex is in maintenance mode.
What is Vue Router? Vue Router is the official routing library for Vue. It enables navigation between different components (pages) in a Single‑Page Application (SPA). Features include: nested routes, dynamic route matching, named routes, route parameters, query parameters, navigation guards, scroll behavior, and lazy loading of routes. In Vue 3, Vue Router 4 is used with the Composition API.
What are navigation guards in Vue Router? Navigation guards are middleware functions that control navigation. They can be: global guards (beforeEach, beforeResolve, afterEach), per‑route guards (beforeEnter), or in‑component guards (beforeRouteEnter, beforeRouteUpdate, beforeRouteLeave). They are used for authentication checks, logging, redirects, or loading data before entering a route.
What is lazy loading in Vue? Lazy loading (or code splitting) is a technique where JavaScript chunks are loaded only when needed, reducing the initial bundle size. In Vue, you can lazy load route components using dynamic imports: component: () => import('@/views/Home.vue'). In the Composition API, you can lazy load components with defineAsyncComponent(). Lazy loading improves performance by loading only the code required for the current view.
What is Vue's Teleport? Teleport is a built‑in component in Vue 3 that renders content outside the current component's DOM hierarchy. It's useful for modals, tooltips, or any UI that needs to be rendered at the top level of the DOM (e.g., for styling or accessibility reasons). You specify the target container with to prop: <Teleport to="body">...</Teleport>. It keeps the component's logic and data in the original context while rendering elsewhere.
What is Vue's Suspense? Suspense is a built‑in component in Vue 3 that manages async dependencies in the component tree. It shows a fallback (loading state) while nested async components (like those using setup() with await) are being resolved. Suspense is useful for data fetching, lazy loading, and code splitting. It works on both server and client, making SSR hydration smoother.
How does Vue handle performance optimization? Vue provides several performance optimizations: virtual DOM with diffing and batching updates, reactive system with fine‑grained dependencies, computed caching, lazy loading (code splitting), component caching (with <KeepAlive>), shallow reactivity (shallowRef, shallowReactive), tree‑shaking (only import what you use), static hoisting in templates, and runtime compilation vs. build‑time compilation. Use v-once for static content, v-memo for memoized sub‑trees, and the Vue Devtools to profile performance.
What are Vue composables (custom hooks)? Composables are reusable functions that encapsulate reactive logic using the Composition API. They allow you to extract and share logic across components. For example, a useFetch() composable that handles data fetching, loading states, and errors. Composables follow a naming convention: useXxx(). They can use Vue's reactivity functions (ref, reactive, watch, onMounted, etc.) and can be composed together.
What are Vue's v-model and two‑way binding? v-model is a directive that creates a two‑way binding between form inputs and reactive data. It's syntactic sugar for binding the value attribute and listening to the input (or change) event. In Vue 3, v-model can be used on custom components with a defined prop and event (default: modelValue and update:modelValue). You can also have multiple v-model bindings on a component by specifying arguments, e.g., v-model:title="pageTitle".
What are Vue's v-if vs. v-show? v-if is a conditional directive that destroys and re‑creates elements based on the condition. It's better for initial rendering performance when the condition is unlikely to change, but it has a higher toggle cost. v-show toggles the display CSS property, keeping the element in the DOM. It's better for frequent toggling because the element is always rendered. v-if can also be used with v-else-if and v-else for more complex condition chains.
How do you handle events in Vue? You handle events using the v-on directive or its shorthand @. For example: <button @click="handleClick">Click</button>. In the component, you define the handler as a method (Options API) or a function (Composition API). Event modifiers like .stop, .prevent, .self, .once, and .capture can be chained (e.g., @click.stop.prevent). You can also pass arguments: @click="handleClick(id, $event)". Key modifiers like @keyup.enter are also supported.
What are Vue's v-for and key attribute? v-for is used for list rendering: <li v-for="item in items" :key="item.id">{{ item.name }}</li>. The key attribute is required for Vue to track each node's identity for efficient re‑rendering (optimizing updates). Without a unique key, Vue might reuse DOM elements incorrectly, causing bugs. key should be a unique identifier (like an ID). In Vue 3, you can also use v-for with <template> for loops without wrapping elements.
What is the <KeepAlive> component? <KeepAlive> is a built‑in component that caches inactive component instances instead of destroying them. It's useful for preserving state when navigating between views (like tabs or dynamic components). Components wrapped in <KeepAlive> will have their activated and deactivated lifecycle hooks triggered. You can configure it with include and exclude props to control which components are cached.
How does Vue handle error boundaries? Vue doesn't have a built‑in error boundary component like React, but you can handle errors using errorCaptured lifecycle hook or the global app.config.errorHandler. errorCaptured is called when an error is thrown in a descendant component. It allows you to log the error, display a fallback UI, or stop propagation. For async errors, use try/catch in setup() or async functions, or handle them in the catch block.
What are the differences between Vue and React? Vue uses HTML‑based templates with directives (v-if, v-for) and a less steep learning curve. React uses JSX (JavaScript XML) and is more flexible but requires more boilerplate. Vue's reactivity is Proxy‑based and more automated; React uses useState and useEffect with manual dependency tracking. Vue's ref and reactive are simpler than React's hooks. Vue has an official framework (Nuxt) similar to Next for React. Vue's ecosystem is smaller but growing. Performance is comparable, but Vue often has smaller bundle sizes.

⚡ Next.js

App Router? The App Router is the new file‑based routing system introduced in Next.js 13, located inside the app/ directory. It supports nested layouts, server components, loading and error boundaries, and parallel/intercepting routes. It replaces the older Pages Router and enables React Server Components by default, leading to better performance and smaller bundle sizes.
Server Component? Server Components are rendered on the server and never sent to the client. They can be async, access databases directly, and import server‑only packages without increasing the client bundle. They cannot use hooks (useState, useEffect) or browser APIs. They are ideal for fetching data, heavy computations, and static content.
Client Component? Client Components are hydrated on the client and provide interactivity. They can use React hooks, event listeners, browser APIs (localStorage, window), and state management. They are marked with the "use client" directive at the top of the file. They are essential for interactive UI elements like forms, modals, and dynamic updates.
Difference? Server Components reduce the client‑side JavaScript bundle and can directly access backend resources. Client Components provide interactivity and browser‑specific features. The two can be nested freely — a Server Component can import a Client Component, but a Client Component cannot import a Server Component directly (they are passed as children or props).
When to use use client? Use "use client" when the component needs hooks (useState, useEffect, useContext), browser APIs (localStorage, window, document), or third‑party libraries that rely on client‑side execution. It is also needed for event handlers (onClick, etc.) and any interactive UI that needs to respond to user input.
SSR / SSG / ISR / CSR? SSR (Server‑Side Rendering): rendered on the server per request — good for dynamic, personalized content. SSG (Static Site Generation): built at build time — fastest, ideal for static content. ISR (Incremental Static Regeneration): SSG with revalidation — updates static pages without full rebuild. CSR (Client‑Side Rendering): rendered in the browser after initial load — good for highly interactive apps but worse for SEO and performance.
Metadata API? The Metadata API allows you to define <head> elements (title, description, Open Graph, etc.) by exporting a metadata object or using the generateMetadata function from layout.tsx or page.tsx. It supports both static and dynamic metadata, improving SEO and social sharing without needing a separate next/head component.
Route Handler? Route Handlers are API endpoints in the App Router, defined in route.ts files inside the app/ directory. You export GET, POST, PUT, etc. functions that handle HTTP requests. They are the equivalent of API routes in the Pages Router (pages/api/) but with better integration with Server Components.
Middleware? Middleware in Next.js runs before requests are completed. It can be used for authentication checks, redirects, rewrites, adding headers, or logging. Middleware runs on the Edge runtime (fast, lightweight) and can conditionally match specific paths using the matcher configuration in middleware.ts.
Authentication in Next? I used NextAuth.js (now Auth.js) which provides built‑in providers (Google, GitHub, credentials) and session management. It integrates seamlessly with both Server and Client Components. If I weren't using NextAuth, I would implement a custom JWT solution with iron-session or a cookie‑based approach, managing access and refresh tokens manually with middleware for route protection.
Why NextAuth? NextAuth simplifies authentication with built‑in OAuth providers, session handling, and CSRF protection. It works with both JWT and database sessions, has a simple configuration, and supports Server Components out‑of‑the‑box. It also handles refresh tokens and token rotation for OAuth providers.
If no NextAuth? I'd implement a custom solution using iron-session for encrypted, stateless sessions, or use jsonwebtoken with HTTP‑only cookies for JWT. I'd also need to manually handle refresh token rotation, logout, and route protection via middleware. This gives more control but requires more boilerplate.
Dynamic Routing? Dynamic routing is achieved using square brackets in folder names, e.g., app/blog/[slug]/page.tsx. The dynamic segment (slug) is accessible via params in the page component. You can also use catch‑all routes ([...slug]) and optional catch‑all ([[...slug]]) for more complex patterns.
Loading.tsx / Error.tsx? Loading.tsx shows a fallback UI (e.g., a spinner) while the page is loading, automatically wrapped in Suspense. Error.tsx catches errors in a route (including in Server Components) and displays a custom error UI. Both can be defined per route or globally in the root app/ directory.
Suspense / Streaming? Suspense is a React component that shows a fallback while its children are loading. In Next.js, it works with loading.tsx and async components. Streaming sends HTML chunks to the client as soon as they are ready, rather than waiting for the entire page to render. This improves Time‑To‑First‑Byte (TTFB) and perceived performance, especially for slow data fetches.

⚛️ React

Why Virtual DOM? The Virtual DOM is a lightweight in‑memory representation of the real DOM. When state changes, React creates a new Virtual DOM tree, compares it with the previous one (diffing), and then updates only the changed nodes in the real DOM. This batching approach avoids expensive direct DOM manipulations, improving performance and developer experience.
useMemo / useCallback / React.memo? useMemo caches the result of a computation and recomputes only when dependencies change — useful for expensive calculations. useCallback caches a function reference and returns the same function instance across renders unless dependencies change — helps prevent unnecessary re‑renders of child components. React.memo is a higher‑order component that prevents a component from re‑rendering if its props haven't changed (shallow comparison).
Context API? Context provides a way to pass data through the component tree without prop drilling (passing props down multiple levels). It's good for global state like theme, user authentication, or language preferences. However, it can cause unnecessary re‑renders because any component consuming a context will re‑render whenever the context value changes, even if only part of the data changed.
Zustand? Zustand is a lightweight state management library that uses a simple API with minimal boilerplate. It creates stores that can be accessed from any component, and it selectively re‑renders only components that use the updated state. It's more scalable than Context API for complex state and doesn't suffer from the same re‑rendering issues.
TanStack Query (React Query)? TanStack Query is a powerful server‑state management library. It handles fetching, caching, synchronizing, and updating server data. It provides features like background refetching, pagination, infinite queries, optimistic updates, and automatic garbage collection. It abstracts away loading/error states and reduces boilerplate in data‑fetching components.
Why React Query? React Query simplifies data fetching by managing the entire lifecycle of server‑state: caching, background updates, retries, and stale‑while‑revalidate. It reduces network requests, keeps data fresh, and improves user experience. It also handles complex scenarios like pagination and infinite scrolling with built‑in hooks (useInfiniteQuery).
Cache? In React Query, the cache stores fetched data. Each query has a unique key, and the data is stored with metadata like staleTime (how long data is considered fresh) and cacheTime (how long data stays in cache after being unused). The cache automatically updates in the background to keep data fresh without blocking the UI.
invalidateQueries? invalidateQueries is a method that marks cached queries as stale, triggering a background refetch when they are next used. It's typically called after a mutation (e.g., after adding or updating data) to ensure the UI reflects the latest server state. You can invalidate specific queries by key or use a partial match to invalidate multiple related queries.

🦁 NestJS

Why NestJS? NestJS is a progressive Node.js framework that uses TypeScript by default and follows an Angular‑like architecture. It's highly modular, uses Dependency Injection for decoupling, and supports many transports (HTTP, WebSockets, GraphQL, gRPC). It's opinionated but flexible, making it ideal for enterprise‑scale applications. It also has great support for microservices and testing.
Module? A Module is a class decorated with @Module() that organizes the application into cohesive blocks. It groups providers (services), controllers, and imports/exports other modules. Modules help structure the codebase, manage dependencies, and enable lazy loading and feature encapsulation.
Provider? A Provider is any class that can be injected as a dependency. It's marked with @Injectable() and can be a service, repository, factory, or helper. Providers are registered in a module and can be injected into controllers or other providers via the constructor.
Dependency Injection? Dependency Injection is a design pattern where an object's dependencies are provided by an external source (the IoC container) rather than created internally. In NestJS, you simply declare dependencies in the constructor with type annotations, and the framework injects the correct instances. This promotes loose coupling, testability, and easier maintenance.
Controller? A Controller handles incoming HTTP requests. It's decorated with @Controller() and contains route handlers using decorators like @Get(), @Post(), @Put(), @Delete(). Controllers receive requests, validate input, delegate to services, and return responses. They should be thin — business logic belongs in services.
Service? A Service is a class that contains business logic and data access. It's decorated with @Injectable() and injected into controllers or other services. Services are reusable, testable, and separate concerns — making the codebase more maintainable. They often use repositories or ORMs to interact with the database.
Pipe? A Pipe is used to transform or validate data. It operates on the arguments passed to route handlers, either globally or for specific routes. ValidationPipe is a built‑in pipe that uses class-validator to validate DTOs. Pipes can also transform input (e.g., string to integer) before it reaches the controller.
Guard? A Guard determines whether a request should be allowed to proceed to the route handler. It's typically used for authentication and authorization. Guards implement the CanActivate interface and return a boolean. They run before the route handler and can be applied globally, per controller, or per route.
Interceptor? An Interceptor adds additional logic around request/response processing. It can bind extra logic before and after route execution, transform the response, map errors, or log requests. Interceptors implement the NestInterceptor interface and are useful for caching, logging, and response formatting.
Middleware? Middleware in NestJS is similar to Express middleware. It runs before the route handler and has access to the request and response objects. Middleware can modify the request/response, end the request, or call the next middleware. It's useful for logging, CORS, and basic auth checks.
Exception Filter? An Exception Filter handles global error handling. It catches exceptions thrown in the application and returns a structured error response. Filters can be applied globally, per controller, or per route. They allow you to define consistent error formats (e.g., { statusCode, message, timestamp }) across your API.
Lifecycle? The request lifecycle in NestJS follows this order:
1. Middleware (runs first) → 2. Guards (auth) → 3. Interceptors (before route) → 4. Pipes (validation/transform) → 5. Controller → Service (business logic) → 6. Interceptor (after route) → 7. Exception Filter (if any error occurred).
ValidationPipe? ValidationPipe is a built‑in pipe that automatically validates incoming payloads against a DTO (Data Transfer Object) using class-validator decorators. If validation fails, it throws a BadRequestException with detailed error messages. It can be used globally or per route, and it also supports transformation (e.g., stripping extra properties).
DTO? A DTO (Data Transfer Object) is a TypeScript class that defines the shape of data being sent over the network. It specifies which fields are expected, their types, and validation rules (using class-validator decorators). DTOs help enforce type safety, validate input, and document the API.
class-validator? class-validator is a library that uses decorators to validate class properties. Common decorators include @IsString(), @IsEmail(), @IsNumber(), @Min(), @Max(), @Length(), and @IsOptional(). It works seamlessly with NestJS's ValidationPipe to auto‑validate incoming requests.
Swagger? Swagger (OpenAPI) is a tool for auto‑generating API documentation. In NestJS, you use @nestjs/swagger with decorators like @ApiTags(), @ApiOperation(), and @ApiResponse(). The documentation is available at /api and provides an interactive UI for testing endpoints, reducing the need for manual documentation.
WebSocket? WebSocket support in NestJS is provided via Gateways decorated with @WebSocketGateway(). Gateways handle real‑time events, broadcasting messages to connected clients. They can use @SubscribeMessage() to listen for client events and integrate with the same DI system as the rest of the application.
SSE? Server‑Sent Events (SSE) allow the server to push events to the client over a single HTTP connection. In NestJS, you can implement SSE using @Sse() decorator that returns an Observable or an async iterable. It's useful for streaming real‑time updates like notifications, AI responses, or live data feeds.

🌐 REST API

REST? REST (Representational State Transfer) is an architectural style for designing networked applications. It uses stateless, cacheable HTTP communication where resources are identified by URLs and manipulated using standard methods (GET, POST, PUT, PATCH, DELETE). It emphasizes scalability, simplicity, and a uniform interface.
PUT vs PATCH? PUT is used to replace an entire resource. The client must send the complete resource representation, and any missing fields will be set to default or null. PATCH is used for partial updates — it applies a set of changes to the resource, allowing you to update only specific fields without affecting the rest. PUT is idempotent; PATCH is not necessarily idempotent.
POST vs PUT? POST is used to create a new resource. It is not idempotent — multiple identical POST requests will create multiple resources. PUT is used to update or replace a resource at a specific URI. It is idempotent — multiple identical PUT requests have the same effect as a single one.
Status Codes? 200 OK, 201 Created, 204 No Content, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 500 Internal Server Error. Each indicates the outcome of the request — 2xx for success, 4xx for client errors, 5xx for server errors.
Pagination? Pagination splits large datasets into smaller chunks. Offset‑based uses ?page=2&limit=20 — simple but suffers from performance issues with large offsets. Cursor‑based uses a pointer (e.g., ?cursor=abc123&limit=20) — more scalable and consistent, ideal for real‑time feeds. I use offset for simple cases and cursor for large or frequently updated data.
Filtering? Filtering allows clients to request a subset of resources. Common patterns include ?status=active for exact match, ?price[gte]=100 for range queries, or ?q=searchTerm for text search. I implement filtering using query parameters and Prisma's where clause to build dynamic filters safely.
Sorting? Sorting orders the returned resources. I use the sort parameter with format ?sort=field:direction (e.g., ?sort=createdAt:desc). Multiple fields can be supported using comma separation, e.g., ?sort=createdAt:desc,name:asc. This maps directly to Prisma's orderBy clause.
Versioning API? API versioning handles breaking changes without disrupting existing clients. Common strategies: URL versioning (/api/v1/users) — most visible and easy to implement; header versioning (e.g., Accept-Version: v1) — cleaner URLs but less discoverable; query parameter versioning (?version=1). I prefer URL versioning for its clarity and simplicity.
Rate Limiting? Rate limiting controls the number of requests a client can make within a time window, preventing abuse, brute‑force attacks, and server overload. In NestJS, I use @nestjs/throttler to apply rate limits globally or per endpoint. I also track limits by IP or user ID, returning 429 Too Many Requests when exceeded.

🗄️ Database

SQL vs NoSQL? SQL databases (PostgreSQL, MySQL) are schema‑based, support ACID transactions, and excel at complex joins and relational data. NoSQL databases (MongoDB, Cassandra) are schema‑less, horizontally scalable, and optimized for high write throughput, logs, and unstructured data. Choose SQL for financial/transactional apps, NoSQL for rapid prototyping or large‑scale analytics.
Why PostgreSQL? PostgreSQL is an object‑relational database with strong ACID compliance, advanced features like JSONB and full‑text search, and excellent performance for complex queries. It's reliable, has a mature ecosystem, and supports extensions (PostGIS, pgvector). It's my default choice for transactional, relational data.
Why MongoDB? MongoDB is a document‑oriented NoSQL database that stores data as JSON‑like documents. It's schema‑flexible, easy to scale horizontally, and supports high write loads. I use it for logging, analytics, and when the data structure is not fixed or requires rapid iteration.
When is MongoDB better? MongoDB is better when the schema is unpredictable or frequently changing, when you need high write throughput, when you're storing unstructured data (logs, social media posts), or when you need to scale horizontally across multiple nodes. It's also easier for developers to get started with because there's no rigid schema.
Normalization? Normalization is the process of organizing data to reduce redundancy and improve data integrity. It breaks large tables into smaller, related tables and defines relationships (one‑to‑many, many‑to‑many). Normalization reduces duplication but can increase query complexity due to joins. Denormalization (adding redundancy) is sometimes used for read performance.
Index? An index is a data structure that improves the speed of data retrieval operations (SELECT queries) at the cost of additional storage and slower writes. PostgreSQL uses B‑tree indexes by default, but also supports GIN, GiST, and hash indexes. Indexes should be created on columns used in WHERE, JOIN, and ORDER BY clauses.
Composite Index? A composite index is an index on multiple columns. The order of columns matters — the index is most efficient for queries that filter on the first column(s). For example, an index on (created_at, status) is good for queries filtering by created_at and optionally status, but less efficient for queries filtering only by status.
Transaction? A transaction is a sequence of operations that are executed as a single atomic unit. Either all operations succeed and are committed, or none are applied and the transaction is rolled back. Transactions are essential for maintaining data consistency in operations like transferring money, updating multiple tables, or any multi‑step process where partial failure is unacceptable.
ACID? Atomicity: transactions are all‑or‑nothing. Consistency: transactions maintain valid state. Isolation: concurrent transactions don't interfere with each other (controlled by isolation levels like Read Committed, Repeatable Read). Durability: committed data persists even after system failure. ACID guarantees reliable, predictable database behavior.
JOIN? A JOIN combines rows from two or more tables based on a related column (typically a foreign key). Common types: INNER JOIN (only matching rows), LEFT JOIN (all left rows + matching right), RIGHT JOIN, and FULL OUTER JOIN. JOINs are used to normalize data and retrieve related information from multiple tables in a single query.
Foreign Key? A foreign key is a column that references the primary key of another table. It establishes a relationship between tables and enforces referential integrity — ensuring that values in the foreign key column exist in the referenced table. Foreign keys help maintain data consistency and prevent orphaned records.
N+1 Problem? The N+1 problem occurs when an ORM executes 1 query to fetch a list of parent records, and then N additional queries to fetch the children for each parent. This leads to poor performance. In Prisma, I solve it using include to eager‑load relations in a single query, or using select with nested fields. For more control, I use raw SQL or batch loading techniques.

🔧 Prisma

Why Prisma? Prisma is a modern ORM that provides type‑safe database access, auto‑generated TypeScript types, and a fluent API. It makes migrations easy, supports relation management, and works with both SQL and NoSQL databases. It's more intuitive than traditional ORMs (like Sequelize) and reduces boilerplate while maintaining performance.
Migration? A migration in Prisma is a declarative way to evolve your database schema. You modify the schema.prisma file, then run prisma migrate dev to generate and apply SQL migrations. Prisma tracks migration history, making it easy to rollback or apply changes in different environments.
Prisma Client? Prisma Client is an auto‑generated query builder that provides a fully type‑safe API for reading and writing data. It's generated based on your schema and offers methods like findMany, create, update, delete, and transaction. It includes autocomplete and type checking, catching errors at compile time.
Prisma Relation? Relations in Prisma are defined in the schema using @relation attributes. They can be one‑to‑one, one‑to‑many, or many‑to‑many. Prisma manages foreign keys and join tables automatically. You can query relations using include or select, and the client keeps relations type‑safe.
Transaction? Prisma supports transactions via $transaction method, which runs multiple operations atomically. You can pass an array of queries or a callback that returns a Promise. All operations succeed or all fail, ensuring data consistency. Transactions also support isolation levels and can be nested.
Prisma Middleware? Prisma Middleware are functions that run before or after Prisma Client operations. They can be used for logging, input validation, soft deletes, or adding timestamps automatically. For example, I use middleware to automatically set createdAt and updatedAt fields, or to filter out soft‑deleted records globally.

🔐 Authentication

JWT? JWT (JSON Web Token) is a compact, URL‑safe token that encodes claims (user ID, roles, etc.) as a JSON object. It consists of three parts: header (algorithm), payload (data), and signature (verifies integrity). The signature ensures the token hasn't been tampered with. JWT is stateless, making it ideal for distributed systems.
Access Token? An Access Token is a short‑lived JWT (typically 15‑30 minutes) used to authorize API requests. It's included in the Authorization header as a Bearer token. Because it expires quickly, the risk of misuse is limited if it's compromised. Access tokens are stateless and don't require database lookups for verification.
Refresh Token? A Refresh Token is a long‑lived token (days or weeks) used to obtain new Access Tokens without re‑authentication. It's stored securely (HttpOnly cookie or server‑side) and is often stored in the database (hashed) to allow revocation. When the Access Token expires, the client sends the Refresh Token to get a new pair, enhancing user experience (no need to re‑login frequently).
Why Refresh Token? Refresh Tokens balance security and usability. If an Access Token is stolen, it expires quickly, limiting damage. Refresh Tokens are more secure because they're stored in HttpOnly cookies and can be revoked server‑side. They also support "remember me" sessions without exposing user credentials.
Where to store token? Best practice is to store tokens in HttpOnly, Secure, SameSite=Strict cookies. This protects against XSS and CSRF. If you must use client‑side storage, use memory (not localStorage/sessionStorage) because they are vulnerable to XSS. For Refresh Tokens, always use HttpOnly cookies.
Cookie vs LocalStorage? Cookies with HttpOnly flag are not accessible via JavaScript, protecting against XSS. LocalStorage is accessible via JavaScript, making it vulnerable to XSS attacks. Cookies also have built‑in expiration and send automatically with requests, while localStorage requires manual injection. For sensitive tokens, always prefer HttpOnly cookies.
HttpOnly? HttpOnly is a cookie flag that prevents JavaScript from accessing the cookie. This is a critical security measure against Cross‑Site Scripting (XSS) attacks, as malicious scripts cannot steal the token. The cookie is only sent to the server via HTTP/HTTPS requests, not via client‑side code.
SameSite? SameSite is a cookie attribute that controls whether the cookie is sent with cross‑site requests. SameSite=Strict only sends the cookie for same‑site requests. SameSite=Lax sends it for top‑level navigation (e.g., following a link). SameSite=None allows cross‑site requests but requires Secure flag. It's a strong defense against CSRF attacks.
CSRF? CSRF (Cross‑Site Request Forgery) is an attack where a malicious website tricks a user's browser into making an unwanted request to a site where they're authenticated. Mitigations include SameSite cookies, anti‑CSRF tokens (sent in forms/headers), and requiring confirmation for sensitive actions. In APIs with JWT, CSRF is less of an issue if tokens are sent in headers.
XSS? XSS (Cross‑Site Scripting) is an attack where malicious scripts are injected into a trusted website, executing in the user's browser. It can steal tokens, session data, or perform actions on behalf of the user. Mitigations include input sanitization, output encoding, Content Security Policy (CSP), and using HttpOnly cookies to protect tokens.
OAuth? OAuth is an open standard for authorization that allows users to grant third‑party applications limited access to their resources without exposing credentials. It works via token issuance and is used in "Login with Google" flows. OAuth 2.0 is the most common version, using flows like Authorization Code (for server‑side apps) and Implicit (for client‑side).
Google Login? Google Login uses OAuth 2.0. The user is redirected to Google, authenticates, and grants permissions. Google returns an authorization code to the callback URL. The backend exchanges this code for an access token and ID token. The ID token contains the user's email and profile, which the backend uses to create or retrieve a user account, then issues its own JWT session.

🐳 Docker

Docker? Docker is a containerization platform that packages applications and their dependencies into lightweight, portable containers. Containers run consistently across any environment, solving the "works on my machine" problem. Docker enables easy deployment, scaling, and isolation of microservices.
Image? A Docker image is a read‑only template that contains the application code, runtime, libraries, and dependencies needed to run the container. Images are built from a Dockerfile and are stored in registries (e.g., Docker Hub). They are the blueprint for creating containers.
Container? A container is a runnable instance of a Docker image. It's isolated from the host system but shares the host kernel, making it lightweight compared to virtual machines. Containers have their own filesystem, network interface, and process space, and can be started, stopped, or removed as needed.
Dockerfile? A Dockerfile is a text file with instructions for building a Docker image. Common instructions: FROM (base image), WORKDIR (working directory), COPY (copy files), RUN (execute commands), EXPOSE (declare ports), and CMD (default command). It defines the entire build process.
docker compose? Docker Compose is a tool for defining and running multi‑container Docker applications. It uses a docker-compose.yml file to configure services, networks, and volumes. For example, you can define your app, PostgreSQL, Redis, and Nginx together, and start them all with a single docker-compose up command.
Multi‑stage Build? Multi‑stage builds reduce final image size by using multiple FROM statements in a single Dockerfile. Each stage is independent, and you can copy artifacts from earlier stages. For example, a stage to build the app (with full toolchain) and a smaller stage to run it (only production dependencies). This keeps the production image lean.
Volume? A volume is a persistent storage mechanism in Docker that survives container restarts. It's used to store data that needs to be preserved (e.g., database files, user uploads). Volumes can be mounted to specific paths in the container and can be managed independently from the container lifecycle.
Port Mapping? Port mapping (or port publishing) maps a container port to a host port using the -p flag, e.g., -p 3000:3000. This allows external traffic to reach services inside the container. Without port mapping, services are only accessible from within the container network.

📦 Git

Git Flow? Git Flow is a branching model for managing releases and features. It defines a main branch (production), a develop branch (integration), feature branches (new features), release branches (preparing releases), and hotfix branches (emergency fixes). It provides structure but can be complex for smaller teams.
merge? Merging combines two branches by creating a new commit that has both branches as parents. It preserves the complete history and is a safe, non‑destructive operation. Merge commits can make history cluttered, but they clearly show where branches converged and are easier to reverse.
rebase? Rebasing rewrites commit history by moving or combining commits to a new base. It creates a linear, clean history by reapplying commits one by one. It's often used before merging to avoid merge commits. However, rebasing is destructive — you should never rebase commits that have been pushed to a shared branch, as it can cause conflicts for other developers.
cherry‑pick? Cherry‑picking allows you to apply a specific commit from one branch to another without merging the entire branch. It's useful for hotfixes or selectively porting changes. However, it can create duplicate commits and conflicts if used improperly.
stash? Stashing temporarily saves uncommitted changes in a safe place, reverting the working directory to the last commit. It allows you to switch branches or pull updates without committing incomplete work. Later, you can reapply the stash (git stash pop or git stash apply).
conflict? A conflict occurs when two branches have made changes to the same part of the same file, and Git cannot automatically resolve the discrepancy. Conflicts must be resolved manually by editing the file to choose which changes to keep. After resolving, you mark it as resolved (git add) and complete the merge/rebase.

🛡️ Security (OWASP)

OWASP Top 10? The OWASP Top 10 is a list of the most critical security risks to web applications, updated periodically. Key risks include: Broken Access Control, Cryptographic Failures, Injection (SQL, XSS), Insecure Design, Security Misconfiguration, Vulnerable Components, Identification Failures, Data Integrity Failures, Logging/Monitoring Failures, and Server‑Side Request Forgery.
XSS? Cross‑Site Scripting (XSS) is an attack where malicious scripts are injected into web pages viewed by other users. It can steal cookies, session tokens, or perform actions on behalf of the user. Mitigations: input sanitization (escape special characters), output encoding (HTML‑encode dynamic content), Content Security Policy (CSP), and using HttpOnly cookies for sensitive tokens.
SQL Injection? SQL Injection occurs when user input is improperly concatenated into SQL queries, allowing attackers to execute arbitrary SQL commands. Prevention: use parameterized queries (Prisma does this automatically), use an ORM that escapes input, and validate/whitelist input. Never concatenate user input directly into SQL strings.
CSRF? Cross‑Site Request Forgery (CSRF) tricks the user's browser into making an unwanted request to a site where they're authenticated. Prevention: SameSite cookies (Strict/Lax), anti‑CSRF tokens (random values sent in forms/headers), and double‑submit cookies. For APIs using JWT in headers, CSRF is naturally mitigated because tokens are not automatically sent with cross‑site requests.
Rate Limiting? Rate limiting restricts the number of requests a client can make within a time period. It prevents brute‑force attacks (password guessing), DoS (Denial of Service), and API abuse. I implement it using @nestjs/throttler, storing counts in Redis for distributed systems. I return 429 Too Many Requests when limits are exceeded.
Password Hashing? Password hashing converts passwords into one‑way hashes for secure storage. I use bcrypt, which is a computationally expensive, salted hash algorithm. The salt (random string) prevents rainbow table attacks. Never store plaintext passwords, and use a slow hashing algorithm (bcrypt, Argon2) to resist brute‑force attacks.

🧠 Architecture

MVC? MVC (Model‑View‑Controller) is a design pattern that separates concerns into three layers: Model (data and business logic), View (UI representation), and Controller (handles user input, updates Model and View). In backend applications, the View is often replaced by JSON responses or templates, and the Controller orchestrates between services and responses.
Layered Architecture? Layered architecture organizes code into horizontal layers, each with a specific responsibility. Common layers: Presentation (API controllers), Business Logic (services), Data Access (repositories/DAOs), and Database. This separation improves maintainability, testability, and allows swapping implementations (e.g., changing databases) without affecting other layers.
Repository Pattern? The Repository pattern abstracts data access behind a collection‑like interface. It acts as a bridge between the domain and data mapping layers, providing a consistent API for querying and persisting objects. It makes it easier to switch databases, mock data for testing, and centralize query logic.
SOLID? Single Responsibility: a class should have only one reason to change. Open/Closed: open for extension, closed for modification. Liskov Substitution: subclasses should be substitutable for their base classes. Interface Segregation: clients shouldn't depend on interfaces they don't use. Dependency Inversion: depend on abstractions, not concretions. Following SOLID makes code more maintainable, testable, and scalable.
Design Patterns used? Singleton: ensuring a class has only one instance (PrismaClient, Logger). Factory: creating objects without specifying the exact class (service factories for different environments). Observer: event‑driven architecture for notifications and real‑time updates. Strategy: encapsulating algorithms (e.g., different payment processors or authentication strategies) that can be swapped at runtime.
Why Dependency Injection? Dependency Injection decouples classes from their dependencies, making the code more modular, testable, and maintainable. Instead of a class creating its own dependencies (tight coupling), dependencies are provided from the outside (loose coupling). This allows for easier mocking in tests and swapping implementations without changing the class itself.
